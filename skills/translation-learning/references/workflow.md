# Workflow

## 总原则

先判断，再翻译；先分流，再组织输出。

这个 skill 的核心价值不在于“会翻译”，而在于先理解用户到底需要哪一种翻译帮助。

## 第一步：任务分流

先把请求归入以下三类：

### `direct_translation`

满足任一条件时优先考虑：

- 用户只是在问“这句怎么翻”“这句什么意思”“帮我翻译一下”
- 请求里只有原文，没有明显的用户译文尝试
- 用户目标明显是获得可用译文，而不是接受批改

### `translation_diagnosis`

满足任一条件时优先考虑：

- 用户明确给出自己的译文，并询问“对不对”“自然吗”“帮我改一下”
- 用户提供原文和另一段目标语言文本，语义上明显是在比较自己的翻译和标准表达
- 用户明确表达了批改、纠错、润色、改写、诊断等意图

### `clarification_needed`

出现以下情况时进入澄清：

- 无法可靠区分哪一段是原文、哪一段是用户自己的译文
- 无法判断方向是 `zh_to_en` 还是 `en_to_zh`
- 用户只给出一句话，但意图可能是翻译、解释、改写或纠错中的多种之一，且风险较高

不要为了追求“直接回答”而做高风险猜测。

## 第二步：信息抽取

尽量从自然语言请求中抽取以下信息：

- `source_text`
- `direction`
- `user_translation`
- 用户是否希望：
  - 只给答案
  - 给答案并解释
  - 重点纠错
  - 给学习建议

### 抽取原则

- 优先依赖用户显式给出的文本边界
- 能从语言本身判断方向时，直接判断
- 只在证据足够时才把某段内容认定为 `user_translation`
- 当用户只是复述目标语言句子而不是声称“这是我写的”，不要自动当作诊断请求

## 第三步：最小澄清

如果核心信息不够，先问一个最小问题。

好澄清问题的特征：

- 一次只问一个最关键的问题
- 尽量让用户补最少的信息
- 不重复询问上下文已经足够推断的内容

示例：

- “你是想让我直接翻译这句，还是帮你看你自己的译文对不对？”
- “这里哪一句是原文，哪一句是你自己写的译文？”
- “你是想翻成英文，还是翻成中文？”

## 第四步：执行流程

### 直接翻译流程

1. 识别原文和方向
2. 生成 `standard_translation`
3. 判断是否需要单独给出 `natural_translation`
4. 判断是否需要补一句简短说明
5. 控制回答长度，避免把简单翻译请求讲成长篇教学

### 批改诊断流程

1. 识别原文、方向和用户译文
2. 生成 `standard_translation`
3. 生成 `natural_translation`
4. 对用户译文做问题诊断
5. 输出问题清单和修改建议
6. 补充学习反馈

## 第五步：输出收束

### 简单请求

- 以结果为主
- 说明从简
- 不自动展开过多规则

### 诊断请求

- 先给正确版本
- 再给问题说明
- 最后给学习反馈

### 多信息请求

即使内容较多，也要分层组织：

- 结果
- 诊断
- 建议

不要把译文和评论混在一段里。
