# Output Patterns

## 总原则

先分清结果和评论。

- 译文是结果
- 诊断、建议、反馈是评论

不要把它们混在一起，导致用户看不清哪个才是最终表达。

## 直接翻译型

适用场景：

- 用户只想知道怎么翻
- 用户没有提供自己的译文

推荐顺序：

1. `standard_translation`
2. 如有必要，给 `natural_translation`
3. 如有必要，补一句简短说明

### 输出要求

- 默认短答
- 不自动展开长篇规则
- 只有在自然表达和标准译文有明显差异时，才额外补充

## 批改诊断型

适用场景：

- 用户提供了自己的译文
- 用户明确希望判断对错、自然度或改写方式

推荐顺序：

1. `standard_translation`
2. `natural_translation`
3. `diagnosis_items`
4. 修改建议
5. `learning_feedback`

### 输出要求

- 先给正确版本
- 再解释问题
- 尽量列全主要问题，但按优先级组织
- 不要把诊断讲得过轻，导致缺少学习价值

## 澄清型

适用场景：

- 原文和用户译文分不清
- 翻译方向不清楚
- 请求意图无法可靠判断

推荐顺序：

1. 一个最小澄清问题
2. 不提前输出高风险猜测

### 输出要求

- 一次只问一个最关键的问题
- 尽量缩短澄清路径
- 能从上下文推断的就不要问
